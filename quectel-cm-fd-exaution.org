* Context
We observed periodic LTE/5G connectivity failures while using *quectel-CM*.
System logs show:

- ~errno:24 (Too many open files)~
- ~Bad file descriptor / No support for INET~
- udhcpc failing to open sysfs nodes:
  - ~/sys/class/net/wwan0/qmi/raw_ip~
  - ~/sys/class/net/wwan0/link_state~

Root cause: quectel-CM hits its soft file descriptor limit (1024), leading to failures in network management.

* Quick Diagnosis

** Get the quectel-cm PID
#+BEGIN_SRC sh :results output replace :exports both
  systemctl show -p MainPID --value quectel-cm
#+END_SRC

** Count open file descriptors (FDs) for the service
#+BEGIN_SRC sh :results output replace :exports both :dir /sudo::
  pid=$(systemctl show -p MainPID --value quectel-cm)
  echo "PID=$pid"
  sudo ls -1 /proc/$pid/fd | wc -l
#+END_SRC

** Check open files limits for the process
#+BEGIN_SRC sh :results output replace :exports both :dir /sudo::
  pid=$(systemctl show -p MainPID --value quectel-cm)
  cat /proc/$pid/limits | egrep -i 'open files'
#+END_SRC

** Inspect recent quectel-CM logs
#+BEGIN_SRC sh :results output replace :exports both
  tail -n 80 /var/log/quectel-CM.log
#+END_SRC

** Inspect systemd status logs for quectel-cm
#+BEGIN_SRC sh :results output replace :exports both :dir /sudo::
  sudo systemctl status quectel-cm.service --no-pager -l
#+END_SRC

* Mitigation 1: Raise FD limit (systemd LimitNOFILE)
This buys stability even if there is a slow FD leak.

** Create systemd override
#+BEGIN_SRC sh
  sudo systemctl edit quectel-cm.service
#+END_SRC

Paste the following into the editor:

#+BEGIN_SRC ini
[Service]
LimitNOFILE=65536
#+END_SRC

** Reload and restart service
#+BEGIN_SRC sh
  sudo systemctl daemon-reload
  sudo systemctl restart quectel-cm.service
#+END_SRC

** Verify new limits and FD count after restart
#+BEGIN_SRC sh
  pid=$(systemctl show -p MainPID --value quectel-cm)
  echo "PID=$pid"
  cat /proc/$pid/limits | egrep -i 'open files'
  sudo ls -1 /proc/$pid/fd | wc -l
#+END_SRC

* Mitigation 2: Provide udhcpc default.script at expected path
quectel-CM logs indicate it expects:

~/usr/share/udhcpc/default.script~

but it is missing (errno 2). This can prevent correct route/DNS setup.

** Install default.script
Adjust the source path if your quectel-CM source directory differs.

#+BEGIN_SRC sh
  sudo mkdir -p /usr/share/udhcpc
  sudo cp /home/nvidia/Downloads/quectel-CM/default.script /usr/share/udhcpc/default.script
  sudo chmod +x /usr/share/udhcpc/default.script
#+END_SRC

** Restart quectel-cm
#+BEGIN_SRC sh
  sudo systemctl restart quectel-cm.service
#+END_SRC

* Monitoring: Track FD growth over time
If FD count keeps increasing steadily, it's likely a leak in quectel-CM/udhcpc flow.

#+BEGIN_SRC sh
  pid=$(systemctl show -p MainPID --value quectel-cm)
  watch -n 30 "date; sudo ls -1 /proc/$pid/fd | wc -l"
#+END_SRC

* Optional Hardening: Daily restart timer (pragmatic)
If the FD leak persists and you need production stability, schedule a controlled restart.

** Create timer unit
#+BEGIN_SRC sh :results output replace :exports both :dir /sudo::
  sudo tee /etc/systemd/system/quectel-cm-restart.timer >/dev/null <<'EOF'
  [Unit]
  Description=Daily restart of quectel-cm to avoid FD leak

  [Timer]
  OnCalendar=*-*-* 04:15:00
  Persistent=true

  [Install]
  WantedBy=timers.target
  EOF
#+END_SRC

** Create service unit
#+BEGIN_SRC sh :results output replace :exports both :dir /sudo::
  sudo tee /etc/systemd/system/quectel-cm-restart.service >/dev/null <<'EOF'
  [Unit]
  Description=Restart quectel-cm

  [Service]
  Type=oneshot
  ExecStart=/bin/systemctl restart quectel-cm.service
  EOF
#+END_SRC

** Enable timer
#+BEGIN_SRC sh :results output replace :exports both :dir /sudo::
  sudo systemctl daemon-reload
  sudo systemctl enable --now quectel-cm-restart.timer
#+END_SRC

** Verify timer is active
#+BEGIN_SRC sh :results output replace :exports both :dir /sudo::
  systemctl list-timers --all | grep -i quectel
#+END_SRC

Ensure eth1 is primary and wwan0 is fallback by metric.
